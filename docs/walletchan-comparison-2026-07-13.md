Both systems share the same core trick: never-deployed `TxSimulator` runtime bytecode injected at the user's own address via `eth_call` `stateOverride.code`, so `address(this) == from` and downstream contracts see `msg.sender == from`. walletchan is the original; viem-tx-sim was inspired by it and rebuilt the domain as a library. walletchan is a wallet product (UI, metadata, and pricing layers sit above its simulation core); viem-tx-sim is a simulation library only — the comparison below sticks to the shared simulation layer.

### Prose highlights

**Asset-type coverage.** Both detect ERC-20 via `balanceOf(address)` and native via `address(this).balance` (viem-tx-sim also via arbitrary-account probes). Both are blind to outgoing ERC-1155 and to ERC-1155 balances generally (the two-arg `balanceOf(address,uint256)` never matches the one-arg probe). Both capture received ERC-721/1155 identity through receiver hooks and an Enumerable walk plus post-state metadata; walletchan adds the `nextTokenId()` counter walk (Uniswap V4 positions) and captures always-on, while viem-tx-sim's capture is opt-in and costs nothing unless requested ([README](https://github.com/frontier159/viem-tx-sim/blob/2db82a8/README.md) documents V4-style counter mints as a known gap).

**Batch modes.** walletchan distinguishes atomic (ERC-7821/7702) from non-atomic (plain EOA) batches; the non-atomic path runs `eth_simulateV1` and bytecode injection **concurrently** and merges (ERC-20/native from v1 transfer logs, NFTs and the revert verdict from the bytecode path — Alchemy's `eth_simulateV1` was observed to false-positive-revert valid approve+swap batches, [txSimulation.ts:2485-2519](https://github.com/apoorvlathey/walletchan/blob/470c2767f8c4b35b399b1e97fc00dd9d1ab9a070/apps/extension/src/chrome/txSimulation.ts#L2485-L2519)). viem-tx-sim has a single batch model: sequential calls in one frame, atomic revert semantics.

**Revert reporting** is the largest quality gap in viem-tx-sim's favor: walletchan reduces a revert to a boolean; viem-tx-sim returns the revert payload, decodes it, and pinpoints the failing call.

**Robustness hardening.** Both gas-cap contract probes and checksum-normalize addresses. Beyond that, the ledger tilts toward viem-tx-sim: walletchan's NFT capture stores at slot 0 under an empty-storage assumption that is false for smart accounts and 7702-delegated EOAs ([TxSimulator.sol:69-72](https://github.com/apoorvlathey/walletchan/blob/470c2767f8c4b35b399b1e97fc00dd9d1ab9a070/apps/contracts/src/utils/TxSimulator.sol#L69-L72)), copies `tokenURI` returndata unbounded ([TxSimulator.sol:523-537](https://github.com/apoorvlathey/walletchan/blob/470c2767f8c4b35b399b1e97fc00dd9d1ab9a070/apps/contracts/src/utils/TxSimulator.sol#L523-L537)), and lacks the EIP-7623 floor in its gas math ([TxSimulator.sol:216-226](https://github.com/apoorvlathey/walletchan/blob/470c2767f8c4b35b399b1e97fc00dd9d1ab9a070/apps/contracts/src/utils/TxSimulator.sol#L216-L226)).

## B. Implementation differences between common features

### B1. Ghost-bytecode injection (the `eth_call`)

Both call `client.call({ account: from, to: from, data: encoded, stateOverride: [{ address: from, code: BYTECODE, ... }] })`.

### B3. Balance snapshots / checkpoints & attribution math

walletchan: one `balanceOf` sweep before, one after the whole batch, signed net delta per candidate, non-zero entries compacted in Solidity ([TxSimulator.sol:314-365](https://github.com/apoorvlathey/walletchan/blob/470c2767f8c4b35b399b1e97fc00dd9d1ab9a070/apps/contracts/src/utils/TxSimulator.sol#L314-L365)) — no per-call granularity, no record of whether a probe *worked* (revert reads as 0 both sides → delta 0 → dropped).

viem-tx-sim: a checkpoint grid — for each probe, `calls.length + 1` readings at stride `calls.length + 1`, row-major; TS reconstructs `before = row[0]`, `after = row[len]`, `byCall[i] = row[i+1] − row[i]` ([checkpoints.ts:13-65](https://github.com/frontier159/viem-tx-sim/blob/2db82a8/src/internal/checkpoints.ts#L13-L65)). `balanceProbeOk` is per-probe AND across all checkpoints; failed probes go to `unresolved` instead of silently reading 0. Separately, min-balance tracking after every successful call feeds `maxTokenOutflows = before − min` and `maxNativeOutflow` — gross flow, not net ([TxSimulator.sol:477-487](https://github.com/frontier159/viem-tx-sim/blob/2db82a8/contracts/TxSimulator.sol#L477-L487)). The same grid mechanics also carry Permit2 allowance checkpoints ([TxSimulator.sol:501-513](https://github.com/frontier159/viem-tx-sim/blob/2db82a8/contracts/TxSimulator.sol#L501-L513)).

Net effect: walletchan can say "USDC −500 overall"; viem-tx-sim can say "call 0: −500, call 1: +499, probe verified at every step, max intra-batch USDC exposure 500".

### B4. Sequential batch execution & failure semantics

Both loop `to.call{value}(data)` in one frame so earlier state changes are visible to later calls. They diverge on failure:

- **walletchan** `simulateBatch` sets `allSuccess = false` and **keeps executing** remaining calls ([TxSimulator.sol:370-377](https://github.com/apoorvlathey/walletchan/blob/470c2767f8c4b35b399b1e97fc00dd9d1ab9a070/apps/contracts/src/utils/TxSimulator.sol#L370-L377)). Deltas reflect the whole hypothetical sequence including calls after the failure; no failing-call index, no revert payload. (This approximates non-atomic EOA semantics, but the same function serves atomic batches, where continuing describes a sequence that cannot happen on-chain.)
- **viem-tx-sim** halts at the first failure, records `failingCallIndex` and the raw `revertData`, and fill-forwards the last checkpoint value so `byCall` entries from the failing call onward are exactly `0n` ([TxSimulator.sol:424-444](https://github.com/frontier159/viem-tx-sim/blob/2db82a8/contracts/TxSimulator.sol#L424-L444), [529-541](https://github.com/frontier159/viem-tx-sim/blob/2db82a8/contracts/TxSimulator.sol#L529-L541)). Atomic revert-all semantics, test-pinned.

### B6. Revert surfacing

Both split "the transaction would revert" (data) from "the simulation infrastructure failed" (walletchan: `simulationFailed` + message string; viem-tx-sim: typed error classes). walletchan propagates only the boolean; the inner call's returndata is discarded at the Solidity level ([TxSimulator.sol:112](https://github.com/apoorvlathey/walletchan/blob/470c2767f8c4b35b399b1e97fc00dd9d1ab9a070/apps/contracts/src/utils/TxSimulator.sol#L112)), so no decode is possible even in TS, and a batch has no failing index. viem-tx-sim captures returndata in-contract, returns it with `failingCallIndex`, and decodes `Error(string)`/`Panic(uint256)` plus caller-supplied custom-error ABIs, always exposing the 4-byte selector ([simulator.ts:344-364](https://github.com/frontier159/viem-tx-sim/blob/2db82a8/src/internal/simulator.ts#L344-L364)).

### B7. NFT identity capture

Both use the same three-part skeleton walletchan pioneered: recording receiver hooks, an ERC-721 Enumerable `tokenOfOwnerByIndex` walk over `[before, after)` capped at 50/collection, and post-state `tokenURI`/`uri` capture under a 5M budget with a 500k return reserve. The implementations diverge in five behaviors:

1. **Activation.** walletchan's is always-on: hooks write storage on every simulation, and `_readAllReceived()` runs unconditionally at the end of every `simulate`/`simulateBatch` ([TxSimulator.sol:128](https://github.com/apoorvlathey/walletchan/blob/470c2767f8c4b35b399b1e97fc00dd9d1ab9a070/apps/contracts/src/utils/TxSimulator.sol#L128), [185](https://github.com/apoorvlathey/walletchan/blob/470c2767f8c4b35b399b1e97fc00dd9d1ab9a070/apps/contracts/src/utils/TxSimulator.sol#L185), [302-308](https://github.com/apoorvlathey/walletchan/blob/470c2767f8c4b35b399b1e97fc00dd9d1ab9a070/apps/contracts/src/utils/TxSimulator.sol#L302-L308)). viem-tx-sim's is flag-gated on `nftCollections.length > 0` — the OFF path performs no SSTORE and the hooks are effectively pure ([TxSimulator.sol:128-132](https://github.com/frontier159/viem-tx-sim/blob/2db82a8/contracts/TxSimulator.sol#L128-L132), [216-239](https://github.com/frontier159/viem-tx-sim/blob/2db82a8/contracts/TxSimulator.sol#L216-L239)); zero-cost-when-unused is test-pinned ([test/nft.test.ts:51](https://github.com/frontier159/viem-tx-sim/blob/2db82a8/test/nft.test.ts#L51)).
2. **Storage placement.** walletchan stores `receivedNfts` at **slot 0**, documented with the assumption "storage at the user's EOA address starts empty for every eth_call" ([TxSimulator.sol:69-72](https://github.com/apoorvlathey/walletchan/blob/470c2767f8c4b35b399b1e97fc00dd9d1ab9a070/apps/contracts/src/utils/TxSimulator.sol#L69-L72)). That assumption fails for smart accounts and 7702-delegated EOAs, whose real storage persists under a code-only override. viem-tx-sim stores capture state at `keccak256("viem-tx-sim.TxSimulator.nftReceipts")` / `...nftCaptureEnabled")` ([TxSimulator.sol:49-53](https://github.com/frontier159/viem-tx-sim/blob/2db82a8/contracts/TxSimulator.sol#L49-L53), [251-273](https://github.com/frontier159/viem-tx-sim/blob/2db82a8/contracts/TxSimulator.sol#L251-L273)); both dirty-slot directions are regression-tested ([test/nft.test.ts:185-219](https://github.com/frontier159/viem-tx-sim/blob/2db82a8/test/nft.test.ts#L185-L219)).
3. **Counter-based mints.** walletchan adds a `nextTokenId()` snapshot + `ownerOf` walk that catches Uniswap V4 positions ([TxSimulator.sol:456-511](https://github.com/apoorvlathey/walletchan/blob/470c2767f8c4b35b399b1e97fc00dd9d1ab9a070/apps/contracts/src/utils/TxSimulator.sol#L456-L511)). viem-tx-sim has no such layer; V4-style counter mints surface only as count deltas (documented, [README](https://github.com/frontier159/viem-tx-sim/blob/2db82a8/README.md)).
4. **Metadata copy bound.** walletchan staticcalls with a gas budget but lets Solidity's automatic returndata copy charge the outer frame unbounded ([TxSimulator.sol:523-537](https://github.com/apoorvlathey/walletchan/blob/470c2767f8c4b35b399b1e97fc00dd9d1ab9a070/apps/contracts/src/utils/TxSimulator.sol#L523-L537)). viem-tx-sim caps `returndatasize()` at 64 KB in assembly before copying; oversized returns are dropped ([TxSimulator.sol:374-392](https://github.com/frontier159/viem-tx-sim/blob/2db82a8/contracts/TxSimulator.sol#L374-L392); [test/nft.test.ts:224-237](https://github.com/frontier159/viem-tx-sim/blob/2db82a8/test/nft.test.ts#L224-L237)).
5. **Duplicate handling.** walletchan's hooks append unconditionally (`_pushReceived`, [TxSimulator.sol:287-299](https://github.com/apoorvlathey/walletchan/blob/470c2767f8c4b35b399b1e97fc00dd9d1ab9a070/apps/contracts/src/utils/TxSimulator.sol#L287-L299)); dedup (`_alreadyCaptured`) guards only the two enumeration walks, so the same ERC-1155 `(collection, id)` received twice yields two rows. viem-tx-sim routes everything through `_recordReceipt`, which aggregates ERC-1155 amounts per `(collection, id)` and no-ops repeated ERC-721 ids ([TxSimulator.sol:278-292](https://github.com/frontier159/viem-tx-sim/blob/2db82a8/contracts/TxSimulator.sol#L278-L292); [test/nft.test.ts:241](https://github.com/frontier159/viem-tx-sim/blob/2db82a8/test/nft.test.ts#L241)).

Shared limitations, both sides: received-only (sent NFTs invisible), Enumerable append-order assumption, plain-`_mint` non-Enumerable ERC-721s invisible (for viem-tx-sim also counter-based mints, per item 3). viem-tx-sim additionally documents the Enumerable swap-and-pop under-report when a batch both receives and sends the same plain-`_mint` collection ([README](https://github.com/frontier159/viem-tx-sim/blob/2db82a8/README.md)).

### B9. Permit2 summary

Forging exists on both sides (walletchan unverified/max-amount, viem-tx-sim verified/non-max/nonce-re-checked — B5); measurement exists only in viem-tx-sim. The estimator gates Permit2 work on the singleton actually appearing among candidates/targets (`permit2Involved`, [requirements.ts:89-101](https://github.com/frontier159/viem-tx-sim/blob/2db82a8/src/internal/requirements.ts#L89-L101)) so Permit2-free estimates pay zero extra RPC; the documented caveat is that an unfunded `from` (degraded discovery) can drop a Permit2 reached only through inner calls, silently unmeasuring it ([README](https://github.com/frontier159/viem-tx-sim/blob/2db82a8/README.md)). Out-of-scope on both sides: `PermitBatch`/`permitTransferFrom` grant detection (viem-tx-sim documents possible over-report; walletchan has no measurement to over-report).

**Why viem-tx-sim needs via-IR and walletchan doesn't.** The trigger is `simulate()`'s 7-parameter signature (6 dynamic calldata arrays + 1 address): the compiler-generated calldata *decoder* overflows the EVM's 16-slot stack reach ("Variable dataEnd is 1 slot too deep") — a body-independent failure no helper-extraction refactor can fix while the signature stands ([foundry.toml:8-15](https://github.com/frontier159/viem-tx-sim/blob/2db82a8/foundry.toml#L8-L15)). walletchan's widest signature is 4 params, and its stack pressure lives in function *bodies*, which the `PreSnapshot`/`_postProcessNfts` extractions relieve — hence legacy codegen suffices there. Measured consolation: via-IR shrinks viem-tx-sim's runtime bytecode ~24% vs a legacy build of the same source (~7.9 KB runtime ≈ 15.7 KB hex per `eth_call` payload).

### Snapshot / delta mechanics

- **walletchan**: pre-call `PreSnapshot { balances, nextTokenIds }` per candidate ([314-326](https://github.com/apoorvlathey/walletchan/blob/470c2767f8c4b35b399b1e97fc00dd9d1ab9a070/apps/contracts/src/utils/TxSimulator.sol#L314-L326)); post-call `_computeDeltas` re-reads `balanceOf`, produces signed `int256` net deltas, compacts non-zero entries in-contract ([332-365](https://github.com/apoorvlathey/walletchan/blob/470c2767f8c4b35b399b1e97fc00dd9d1ab9a070/apps/contracts/src/utils/TxSimulator.sol#L332-L365)). All balance reads target `address(this)` only. Native: single `int256 ethDelta = after − before`.
- **viem-tx-sim**: candidate classification (`_snapshotTokens`), per-call allowance/balance/Permit2 checkpoint grids at stride `calls.length + 1` with fill-forward on failure, min-balance gross-outflow tracking, arbitrary-account and native (`address(0)`) balance probes ([394-541](https://github.com/frontier159/viem-tx-sim/blob/2db82a8/contracts/TxSimulator.sol#L394-L541), [564-571](https://github.com/frontier159/viem-tx-sim/blob/2db82a8/contracts/TxSimulator.sol#L564-L571)); delta math lives in TS. `_executeCalls` returns a clean `(bool, uint256, bytes, uint256)` tuple ([394-400](https://github.com/frontier159/viem-tx-sim/blob/2db82a8/contracts/TxSimulator.sol#L394-L400)).
- **Aggregate vs per-call**: walletchan is aggregate-only; viem-tx-sim is per-call with a fill-forward tail so post-failure per-call deltas reconstruct to exactly `0n`.
- **Gross outflow**: viem-tx-sim only — `maxTokenOutflows[i] = before − min`, `maxNativeOutflow = nativeStart − nativeMin` ([150-155](https://github.com/frontier159/viem-tx-sim/blob/2db82a8/contracts/TxSimulator.sol#L150-L155), [468](https://github.com/frontier159/viem-tx-sim/blob/2db82a8/contracts/TxSimulator.sol#L468)).

### Gas budgeting

- walletchan: 100k probe cap on `_tryBalanceOf`/`_tryNextTokenId`/`_tryOwnerOf`/Enumerable walks; metadata `min(gasleft() − 500k, 5M)` with the loop breaking at budget 0 rather than reverting ([36-47](https://github.com/apoorvlathey/walletchan/blob/470c2767f8c4b35b399b1e97fc00dd9d1ab9a070/apps/contracts/src/utils/TxSimulator.sol#L36-L47), [229-238](https://github.com/apoorvlathey/walletchan/blob/470c2767f8c4b35b399b1e97fc00dd9d1ab9a070/apps/contracts/src/utils/TxSimulator.sol#L229-L238), [539-544](https://github.com/apoorvlathey/walletchan/blob/470c2767f8c4b35b399b1e97fc00dd9d1ab9a070/apps/contracts/src/utils/TxSimulator.sol#L539-L544)); regression-tested against a gas-burning `balanceOf` and a keccak-heavy `tokenURI` ([TxSimulator.t.sol:45-93](https://github.com/apoorvlathey/walletchan/blob/470c2767f8c4b35b399b1e97fc00dd9d1ab9a070/apps/contracts/test/TxSimulator.t.sol#L45-L93)).
- viem-tx-sim: 150k probe cap applied through the single `_tryStaticUint` primitive that all four one-word probes route through — one hardening surface instead of four copies ([32](https://github.com/frontier159/viem-tx-sim/blob/2db82a8/contracts/TxSimulator.sol#L32), [548-590](https://github.com/frontier159/viem-tx-sim/blob/2db82a8/contracts/TxSimulator.sol#L548-L590)); the same 5M/500k metadata split; plus the 64 KB returndata bound walletchan lacks ([374-392](https://github.com/frontier159/viem-tx-sim/blob/2db82a8/contracts/TxSimulator.sol#L374-L392)).

### Receiver hooks / ERC-165 / ERC-1271

- **Hooks**: walletchan's are stateful always (push to the slot-0 array, [246-299](https://github.com/apoorvlathey/walletchan/blob/470c2767f8c4b35b399b1e97fc00dd9d1ab9a070/apps/contracts/src/utils/TxSimulator.sol#L246-L299)). viem-tx-sim's are stateful *when capture is enabled* (`if (_captureEnabled()) _recordReceipt(...)`, [216-239](https://github.com/frontier159/viem-tx-sim/blob/2db82a8/contracts/TxSimulator.sol#L216-L239)) and effectively pure otherwise. Both return the standard magic values so safe transfers never revert.
- **ERC-165**: both advertise `0x01ffc9a7`, `0x150b7a02`, `0x4e2312e0` (walletchan [551-556](https://github.com/apoorvlathey/walletchan/blob/470c2767f8c4b35b399b1e97fc00dd9d1ab9a070/apps/contracts/src/utils/TxSimulator.sol#L551-L556); viem-tx-sim [244-247](https://github.com/frontier159/viem-tx-sim/blob/2db82a8/contracts/TxSimulator.sol#L244-L247)). viem-tx-sim has no `fallback()` — unknown selectors revert by design.
- **ERC-1271 — materially different.** walletchan returns the magic value for **any** signature that `ecrecover`s to non-zero, 65-byte only, no `v` normalization ([565-583](https://github.com/apoorvlathey/walletchan/blob/470c2767f8c4b35b399b1e97fc00dd9d1ab9a070/apps/contracts/src/utils/TxSimulator.sol#L565-L583)) — during simulation, a wrong-signer or forged permit signature previews as valid, a false-positive preview vector. viem-tx-sim requires `signer == address(this)`, normalizes `v`, supports EIP-2098 compact signatures ([212-214](https://github.com/frontier159/viem-tx-sim/blob/2db82a8/contracts/TxSimulator.sol#L212-L214), [599-630](https://github.com/frontier159/viem-tx-sim/blob/2db82a8/contracts/TxSimulator.sol#L599-L630)), reproducing real EOA verification semantics. Both replace a real contract wallet's own ERC-1271 during simulation (documented limitation on the viem-tx-sim side).

### NFT / tokenURI machinery

See B7 for the five behavioral divergences. Contract-level summary: walletchan = 3 capture paths (hooks, Enumerable, `nextTokenId`) + unbounded post-state URI capture, slot-0 storage, append-only with walk-side dedup ([246-544](https://github.com/apoorvlathey/walletchan/blob/470c2767f8c4b35b399b1e97fc00dd9d1ab9a070/apps/contracts/src/utils/TxSimulator.sol#L246-L544)). viem-tx-sim = 2 capture paths (hooks, Enumerable) + bounded post-state URI capture, hashed-slot storage, single aggregating record path ([216-292](https://github.com/frontier159/viem-tx-sim/blob/2db82a8/contracts/TxSimulator.sol#L216-L292), [311-392](https://github.com/frontier159/viem-tx-sim/blob/2db82a8/contracts/TxSimulator.sol#L311-L392)).

### Return encoding

- walletchan: positional tuple `(bool, int256, address[], int256[], NftReceived[])`; `NftReceived` carries raw `tokenUriRaw` bytes decoded TS-side.
- viem-tx-sim: one 11-field `SimulationResult` struct (flat checkpoint grids, `observedTokens`, outflow aggregates, `balanceProbeOk`, `permit2Checkpoints`, `nftReceipts`, `failingCallIndex`, `revertData`); `simulateBatchGas` returns a flat 3-tuple; all mirrored 1:1 by the TS ABI with a drift-guard test covering every declared function ([84-96](https://github.com/frontier159/viem-tx-sim/blob/2db82a8/contracts/TxSimulator.sol#L84-L96); [test/abi.test.ts:42-53](https://github.com/frontier159/viem-tx-sim/blob/2db82a8/test/abi.test.ts#L42-L53)).

### Things each contract does that the other doesn't

- **Only walletchan**: `nextTokenId()` counter walk; continue-past-failure batches; in-contract intrinsic/calldata gas; in-contract signed-delta compaction.
- **Only viem-tx-sim**: allowance + Permit2 checkpoint grids; arbitrary-account and native balance probes; per-call grids with fill-forward; gross-outflow tracking; revert-data capture + failing index; signer-bound ERC-1271 with EIP-2098; per-probe ok-tracking (`balanceProbeOk`); flag-gated capture at namespaced hashed slots; bounded metadata returndata copy; receipt aggregation; halt-and-report gas measurement with a failing index; consolidated `_tryStaticUint` probe primitive.
