# Learnings for viem-tx-sim from walletchan (2026-07-12)

Companion to `docs/walletchan-comparison-2026-07-12.md` (citations and side-by-side detail live there). walletchan = https://github.com/apoorvlathey/walletchan at `470c2767`; GitHub citation form `https://github.com/apoorvlathey/walletchan/blob/main/<path>`.

Constraints honored throughout (from CLAUDE.md): RPC-only; viem as sole runtime dependency; pinned invariants — public `simulate()` emits exactly one `eth_call` and zero `eth_createAccessList`; deterministic ordering; the `10^50` non-max sentinel; transaction reverts as result status, infrastructure failures as typed errors. Recommendations that touch generated bytecode or a pinned invariant are flagged; per the plans workflow, each adopted item should become a `plans/` entry before implementation — no plan files are created here.

---

## Recommendations to adopt

### 1. Gas-cap the contract's probe staticcalls — HIGH

- **What walletchan does**: every best-effort probe (`_tryBalanceOf`, `_tryNextTokenId`, `_tryOwnerOf`, Enumerable walk) carries `staticcall{gas: PROBE_GAS_LIMIT}` with `PROBE_GAS_LIMIT = 100_000` (apps/contracts/src/utils/TxSimulator.sol:36-40, 229-238). Regression-tested: `testNativeDeltaSurvivesGasBurningBalanceProbe` deploys a target whose fallback infinite-loops specifically on the `balanceOf` selector and asserts the native delta still comes through (apps/contracts/test/TxSimulator.t.sol:45-66).
- **Why it matters**: our `_tryBalanceOf`/`_tryAllowance` are uncapped (`contracts/TxSimulator.sol:248-254, 265-275`). A hostile or merely pathological `balanceOf` in the candidate list (candidates come from access lists — any contract in the call path can end up there) or in a caller-supplied `balanceQueries` token burns 63/64 of remaining gas per probe. Our probes run once per probe per call boundary, so a couple of burns OOG the whole `eth_call`. Worse, that surfaces as `StateOverrideUnsupportedError` — a *misclassified infrastructure failure* — instead of an unresolved probe. Same vector poisons `discoverErc20s` (one bad candidate kills the token filter) and `estimateRequirements` recon.
- **Adoption shape**: add a gas stipend to `_tryBalanceOf` and `_tryAllowance`. Behavior after: a gas-burning token becomes `ok = false` → the query lands in `unresolved` / the candidate is dropped, and every other probe still resolves. Pick the cap above worst realistic legit reads (proxied tokens with hooks; 100k is walletchan-proven, 150-200k is safer headroom) and add a Foundry/Anvil regression mirroring theirs. No RPC-count or result-shape change; `unresolved` semantics already cover the new failure mode.
- **Flag**: changes `src/generated/txSimulatorBytecode.ts` → plan-gated bytecode regeneration (`pnpm build:contracts`).

### 2. Checksum-normalize `from` (and call params) at the RPC boundary — HIGH

- **What walletchan does**: runs `getAddress()` on `from` and `to` before every simulation, with a documented root cause: viem's `formatStateOverride` checksums override-map keys while `formatTransactionRequest` leaves `from` untouched, and some RPCs (Coinbase's `mainnet.base.org` proxy) reject the mixed-casing request with `-32602` (apps/extension/src/chrome/txSimulation.ts:1026-1034).
- **Why it matters**: we have exactly the asymmetry they patched. `buildStateOverride` normalizes override addresses (`src/internal/simulator.ts:264` via `normalizeAddress`), but the call's `account`/`to` pass `args.from` verbatim (`src/internal/simulator.ts:127-128`). A caller who supplies a lowercase `from` (common: addresses copied from explorers/logs) gets a checksummed override key next to a lowercase `from` — the precise shape that draws `-32602` from casing-sensitive proxies. Result: a spurious `StateOverrideUnsupportedError` on an RPC that actually supports state overrides.
- **Adoption shape**: `normalizeAddress(args.from)` once in `runSimulator` (and the equivalent in `createAccessList` / `readUint256Call` request builders) so tx fields and override keys always agree in casing. Pure input normalization; no invariant touched; one unit test asserting a lowercase `from` produces checksummed `from`/`to`/override-key in the outgoing request.

### 3. Cap `eth_createAccessList` gas independently of `eth_call` gas — HIGH

- **What walletchan does**: two knobs — 10M for `eth_createAccessList` and single-tx `eth_call`, 50M for the batch `eth_call` — because "Alchemy mainnet rejects 30M while accepting ≤ ~10M for eth_createAccessList" while "most RPC providers accept 50M+ for eth_call" (apps/extension/src/chrome/txSimulation.ts:110-136).
- **Why it matters**: our single default (`DEFAULT_SIMULATION_GAS_LIMIT = 16_000_000n`, `src/constants.ts:7`) flows into **every** `eth_createAccessList` request (`src/internal/simulator.ts:205` → `src/internal/rpc.ts:93`). 16M sits above walletchan's empirically observed ~10M Alchemy ceiling for that method — meaning `balanceQueries.forUser`, `discoverErc20s`, `estimateRequirements`, and all slot discovery may throw `AccessListUnsupportedError` against Alchemy-class providers while the simulation `eth_call` itself would have worked.
- **Adoption shape**: clamp or separately default the gas sent on `eth_createAccessList` (e.g. `min(gas, 10M)` or a distinct constant), leaving `eth_call` gas alone. Verify empirically against Alchemy/LlamaRPC before pinning the number (`pnpm test:mainnet` is the natural home). No RPC-count changes; debug events already expose `hasGas` per request for assertion.

### 4. `supportsInterface` (ERC-165) on the ghost — MEDIUM

- **What walletchan does**: advertises ERC-165 + ERC-721-receiver (`0x150b7a02`) + ERC-1155-receiver (`0x4e2312e0`) (apps/contracts/src/utils/TxSimulator.sol:551-556) so contracts that pre-check `supportsInterface` before `safeTransferFrom` proceed.
- **Why it matters**: against the real chain, `from` has no code, so senders skip receiver checks entirely. During simulation `from` *has* code, so a marketplace/router that probes `supportsInterface` before a safe transfer hits our no-fallback revert → false "simulated revert" for a transaction that succeeds on-chain. We already accepted this class of fix for the receiver hooks themselves (plan 015); ERC-165 is the same argument one selector wider.
- **Adoption shape**: add `supportsInterface(bytes4)` returning true for exactly `0x01ffc9a7`, `0x150b7a02`, `0x4e2312e0` (and arguably `0x1626ba7e`-adjacent ERC-1271 detection ids if we find flows that probe it). Keep the no-`fallback()` posture — this is an explicit selector, not interface-faking. Behavior change: safe-transfer flows with ERC-165 pre-checks stop false-reverting.
- **Flag**: bytecode regeneration → plan-gated.

### 5. Permit2 allowance forging (nonce-preserving) — MEDIUM

- **What walletchan does**: computes Permit2's triple-nested `allowance[owner][token][spender]` slot (three nested keccaks from slot 0), reads the current packed value via `eth_getStorageAt`, and overrides expiration + amount to max **while preserving the on-chain 48-bit nonce**, because `permit()` verifies the signed nonce against storage (apps/extension/src/chrome/txSimulation.ts:854-896).
- **Why it matters**: modern swap paths (Uniswap Universal Router, 0x) draw funds through Permit2, whose allowance getter is `allowance(owner,token,spender)` — our ERC-20-selector probing can never discover or forge it. So `estimateRequirements` on a Permit2-routed path reverts in the forged measurement even after we've forged the ERC-20 balance and the ERC-20→Permit2 approval, and reports nothing useful about the Permit2 leg.
- **Adoption shape**: a Permit2-aware override preparer keyed on the canonical singleton address: computed slot math (deterministic, matches our existing `mappingSlot` style in `src/internal/slots.ts:189-200`), one `eth_getStorageAt` (or sentinel-verified `eth_call` read) per pair for the nonce, sentinel-verify the composed slot like everything else. Two wrinkles to resolve in the plan: (a) the packed amount field is `uint160`, and `OVERRIDE_TOKEN_AMOUNT = 10^50` **exceeds `2^160 − 1 ≈ 1.46×10^48`** — a Permit2-specific sub-`2^160`, still-non-max sentinel is required; (b) whether Permit2 allowance drawdown joins the requirements measurement (would need an `allowance(owner,token,spender)`-selector probe variant in the contract). RPC-only and viem-only throughout.
- **Flag**: extends (does not violate) the sentinel doctrine — document the second sentinel next to `OVERRIDE_TOKEN_AMOUNT`. Contract change only if Permit2 probes are added → plan-gated.

### 6. Graceful candidate degradation when access lists are rejected outright — MEDIUM

- **What walletchan does**: if `eth_createAccessList` throws (observed: Alchemy rejecting it when `from` has zero ETH), candidates degrade to `[to]` and the simulation still runs (apps/extension/src/chrome/txSimulation.ts:1106-1109).
- **Why it matters**: we normalize *reverts* to `[]` (`src/internal/rpc.ts:113-119`) and `estimateRequirements` degrades on insufficient-funds messages (`src/internal/requirements.ts:52-55`), but any other provider rejection makes `balanceQueries.forUser`/`discoverErc20s` throw `AccessListUnsupportedError` and produce nothing — even though a `[call.to]`-only candidate set would still catch direct token transfers.
- **Adoption shape**: don't make it silent (walletchan's console.warn is invisible to a library caller; silent narrowing hides missed tokens). Either broaden the insufficient-funds fallback already used by the estimator to discovery, or surface a partial result with an explicit "degraded discovery" marker. Decide in the plan; the invariant "throw typed errors on infra failure" is the tension to resolve deliberately, not by copying walletchan.

### 7. NFT tokenId / tokenURI capture — LOW

- **What walletchan does**: storage-accumulating receiver hooks plus ERC-721 Enumerable and `nextTokenId()` enumeration fallbacks and post-state `tokenURI`/`uri` capture with a metadata gas budget (apps/contracts/src/utils/TxSimulator.sol:246-544). This is the only way to answer "*which* position NFT will I receive from this Uniswap V3/V4 mint" from a pure `eth_call`, and post-state URI capture is genuinely unrepeatable after the call returns.
- **Why it matters**: we currently show NFTs as bare count deltas; wallets built on us must re-derive identity themselves and *cannot* recover post-state metadata at all.
- **Adoption shape if ever pursued**: a new probe family (opt-in, like `balanceQueries`) so the default path stays lean; contract gains the capture machinery; result gains an NFT section. Changes the contract ABI (guarded by `test/abi.test.ts`), the return shape, and bytecode → substantial plan. Low priority because it's additive scope, not a correctness gap in what we already claim to do (README documents token-ID ownership as unmodeled).

### 8. Sub-call gas measurement (`simulateBatchGas` analog) — LOW

- **What walletchan does**: `gasleft()`-based per-call gas inside the ghost, plus 21000 intrinsic and per-byte calldata gas so results are usable as tx gas limits; consumers apply a flat 2× buffer because raw sequential measurement has no headroom for EIP-150's 63/64 forwarding dilution through deep call trees (apps/contracts/src/utils/TxSimulator.sol:198-226; apps/extension/src/chrome/batchGasEstimation.ts:286-304). It exists because dependent calls (swap-after-approve) cannot be `eth_estimateGas`-ed standalone.
- **Why it matters / shape**: real user problem (gas limits for non-atomic 5792 batches), and our contract already runs the sequential loop — but it's outside our asset-change scope and the injected simulator's own probe overhead pollutes `gasleft()` deltas unless measured around a bare loop like theirs (a separate entry point). Note the 2× EIP-150 rationale if this is ever built; until then, out of scope.

## Deliberately NOT adopting

- **walletchan's ERC-1271 semantics.** Theirs returns the magic value for *any* signature that `ecrecover`s to a non-zero address — it never compares the signer to `address(this)` (apps/contracts/src/utils/TxSimulator.sol:565-583). That validates forged signatures during simulation and can preview flows a real broadcast rejects. Ours (recover-and-compare, `v` normalization, EIP-2098 support; `contracts/TxSimulator.sol:93-95, 284-315`) reproduces real EOA verification and stays.
- **Unverified slot discovery.** walletchan takes the first non-proxy storage key from the access list without verifying it backs the value (txSimulation.ts:734-763). A wrong slot silently forges garbage state (packed slots, multi-slot reads, unusual proxies). Our sentinel-write-and-readback verification (`src/internal/probes.ts:77-91`) is strictly safer; keep it, keep `unresolved` for the rest.
- **Automatic 100,000-ETH balance override at `from`** (txSimulation.ts:1181). It makes every preview lie about native affordability: a user who cannot fund `value` sees a clean success. Ours reporting the real insufficient-balance revert is the correct default; callers who want forged funds have `nativeBalanceOverrides`, and `estimateRequirements` forges deliberately.
- **Auto retry-with-forged-state inside `simulate()`** (txSimulation.ts:1118-1128). A conditional second `eth_call` plus per-candidate discovery calls would break the pinned "one `eth_call`, zero access lists" invariant and make RPC cost unpredictable. Forging stays an explicit, separately-billed step (`tokenOverrides.*`).
- **`MAX_UINT256` allowance forging** (txSimulation.ts:848-853). Max allowance suppresses ERC-20 allowance decrements, which is exactly what our measurement observes; this is the failure mode the `10^50` sentinel exists to avoid, and TS already rejects max by design (`src/internal/simulator.ts:297-300`).
- **Continue-past-failure batch execution** (their contract:370-377). It forfeits `failingCallIndex`/`revertData` and would break the pinned zero-tail `byCall` invariant. If genuine non-atomic (keep-broadcasting) semantics are ever wanted, that is an explicit new execution mode via a plan — not a change to the default.
- **`eth_simulateV1` as a second verdict channel** (txSimulation.ts:2189-2520). walletchan's own experience is the argument: they had to hard-code "never trust `eth_simulateV1`'s revert verdict" after Alchemy false-positive-reverted valid batches. A second RPC method also breaks the single-method portability story and the pinned counts. If transfer-log-based discovery is ever attractive (it would see outgoing ERC-1155), it belongs in a separate opt-in helper, never in `simulate()`.
- **ERC-7715 static decode, metadata/pricing enrichment, retry loops, post-confirmation log extraction, per-chain gas-model registry.** Wallet-product concerns (UI freshness, CoinGecko, chain registry). A library consumer composes these; none affect simulation correctness at our layer.
