# Notes for walletchan: behaviors from viem-tx-sim worth borrowing back (2026-07-13)

Hi — [viem-tx-sim](https://github.com/frontier159/viem-tx-sim) is a TypeScript library that grew out of walletchan's bytecode-injection simulation trick (never-deployed `TxSimulator` code placed at the user's own address via `eth_call` state overrides, so `msg.sender` matches reality). Several of walletchan's designs are load-bearing code in viem-tx-sim today — the probe gas caps, the Permit2 nonce-preservation trick, the metadata gas-budget split, the Enumerable walk, the empirically-derived provider gas ceilings — and they held up well under adversarial review. This note goes the other direction: behaviors present in viem-tx-sim but not in walletchan that seem genuinely useful upstream, each verified against walletchan's source before being written down.

Scope: features and behaviors present in **viem-tx-sim** but not in **walletchan** — what each does, why it matters, and a sizing note. Mirror of the inbound analysis kept on the viem-tx-sim side.

Sources: **walletchan** — https://github.com/apoorvlathey/walletchan at `470c2767`; **viem-tx-sim** — https://github.com/frontier159/viem-tx-sim at `2db82a8`. "txSimulation.ts" = `apps/extension/src/chrome/txSimulation.ts`; "walletchan's contract" = `apps/contracts/src/utils/TxSimulator.sol`; "viem-tx-sim's contract" = `contracts/TxSimulator.sol`.

## 1. Feature gaps — candidates for adoption

### 1.1 ERC-1271: compare the recovered signer against `address(this)` — priority: HIGH, size: S

- **What walletchan does today**: the ghost's `isValidSignature` returns the magic value whenever `ecrecover` yields **any** non-zero address — it never compares the signer to `address(this)`, doesn't normalize `v`, and only handles 65-byte signatures ([walletchan's contract:565-583](https://github.com/apoorvlathey/walletchan/blob/470c2767f8c4b35b399b1e97fc00dd9d1ab9a070/apps/contracts/src/utils/TxSimulator.sol#L565-L583)).
- **Why it matters**: during simulation, any well-formed signature from **any** signer validates. A Permit2 `permit()` carrying a wrong-signer or forged signature previews as a clean success, then the real broadcast reverts — a false-positive preview in exactly the flow (permit-based approvals) where the preview matters most. The asymmetry: because the ghost gives the EOA code, Permit2 takes its ERC-1271 branch during simulation but the raw ecrecover-and-compare branch on-chain, so the ghost's ERC-1271 must reproduce EOA semantics exactly or the two paths diverge.
- **Adoption note**: one comparison plus `v` normalization; EIP-2098 compact 64-byte support costs a few more lines. Reference implementation: [viem-tx-sim's contract:212-214](https://github.com/frontier159/viem-tx-sim/blob/2db82a8/contracts/TxSimulator.sol#L212-L214), [599-630](https://github.com/frontier159/viem-tx-sim/blob/2db82a8/contracts/TxSimulator.sol#L599-L630).

### 1.2 Move NFT capture storage off slot 0 — priority: HIGH, size: S

- **What walletchan does today**: `receivedNfts` is declared at slot 0 with the rationale "storage at the user's EOA address starts empty for every eth_call invocation" ([walletchan's contract:69-72](https://github.com/apoorvlathey/walletchan/blob/470c2767f8c4b35b399b1e97fc00dd9d1ab9a070/apps/contracts/src/utils/TxSimulator.sol#L69-L72)).
- **Why it matters**: a `stateOverride.code` entry replaces only the **code** at `from`; the account's real storage persists into the `eth_call`. That's fine for a pristine private-key EOA and false for exactly the accounts the atomic batch path serves — Bankr smart accounts and, per walletchan's own docs, PK/SP EOAs that "now execute multi-call batches atomically via EIP-7702 + ERC-7821" ([_docs/ERC5792.md:50-57](https://github.com/apoorvlathey/walletchan/blob/470c2767f8c4b35b399b1e97fc00dd9d1ab9a070/_docs/ERC5792.md#L50-L57)). Honestly assessed: delegates using ERC-7201 namespaced storage dodge this by construction, but any account keeping state at slot 0 (Safe-style proxies with the singleton there, simple owner-at-slot-0 delegates, third-party delegates the resolver accepts) collides. The blast radius exceeds NFT rows: `_readAllReceived()` runs unconditionally at the end of **every** `simulate`/`simulateBatch` ([128](https://github.com/apoorvlathey/walletchan/blob/470c2767f8c4b35b399b1e97fc00dd9d1ab9a070/apps/contracts/src/utils/TxSimulator.sol#L128), [185](https://github.com/apoorvlathey/walletchan/blob/470c2767f8c4b35b399b1e97fc00dd9d1ab9a070/apps/contracts/src/utils/TxSimulator.sol#L185), [302-308](https://github.com/apoorvlathey/walletchan/blob/470c2767f8c4b35b399b1e97fc00dd9d1ab9a070/apps/contracts/src/utils/TxSimulator.sol#L302-L308)), reads slot 0 as the array length, and allocates `new NftReceived[](len)` — an address-sized value means `len ≈ 2^160` → instant OOG → the *whole* asset preview fails; a small integer instead phantom-fabricates receipts from whatever `keccak256(0)`-adjacent storage holds. walletchan already special-cases ERC-7715 redemptions because injecting over a 7702 account's *code* breaks things ([txSimulation.ts:909-986](https://github.com/apoorvlathey/walletchan/blob/470c2767f8c4b35b399b1e97fc00dd9d1ab9a070/apps/extension/src/chrome/txSimulation.ts#L909-L986)) — this is the same bug's storage-shaped sibling.
- **Adoption note**: ~15 lines — move all capture state to keccak-namespaced slots (ERC-7201 spirit) behind small assembly accessors. Reference: [viem-tx-sim's contract:49-53](https://github.com/frontier159/viem-tx-sim/blob/2db82a8/contracts/TxSimulator.sol#L49-L53) (constants + rationale), [251-273](https://github.com/frontier159/viem-tx-sim/blob/2db82a8/contracts/TxSimulator.sol#L251-L273) (accessors), with dirty-low-slot regressions in both directions ([test/nft.test.ts:185-219](https://github.com/frontier159/viem-tx-sim/blob/2db82a8/test/nft.test.ts#L185-L219)) — that test suite exists because viem-tx-sim hit exactly this collision (a panic `0x41` allocation failure) before namespacing.

### 1.3 Bound the metadata returndata copy — priority: HIGH, size: S

- **What walletchan does today**: `_captureTokenUris` staticcalls each renderer under a well-chosen gas budget ([walletchan's contract:523-537](https://github.com/apoorvlathey/walletchan/blob/470c2767f8c4b35b399b1e97fc00dd9d1ab9a070/apps/contracts/src/utils/TxSimulator.sol#L523-L537)) — but the `(bool ok, bytes memory ret) = ...staticcall{gas: budget}(...)` form makes Solidity copy the **entire** returndata into the calling frame, and that memory-expansion gas is charged *outside* the callee's budget.
- **Why it matters**: a renderer can afford to RETURN on the order of a megabyte within a 5M own-frame budget; the simulator frame then pays again to copy it, frame memory is never freed, and memory expansion is quadratic — a handful of hostile (or just bloated) receipts compound to OOG even the 50M batch call. One adversarial collection in a batch can sink the entire preview through the metadata path, despite `METADATA_GAS_LIMIT` doing its job on the callee side. The existing `testCapturesHeavyOnchainNftMetadata` covers the gas-heavy renderer; the *return-heavy* one slips past it.
- **Adoption note**: check `returndatasize()` before copying and drop oversized returns — an assembly staticcall of ~15 lines. viem-tx-sim caps at 64 KB (generous for real data-URI SVGs): [contract:374-392](https://github.com/frontier159/viem-tx-sim/blob/2db82a8/contracts/TxSimulator.sol#L374-L392), constant + rationale at [44-47](https://github.com/frontier159/viem-tx-sim/blob/2db82a8/contracts/TxSimulator.sol#L44-L47), regression at [test/nft.test.ts:224-237](https://github.com/frontier159/viem-tx-sim/blob/2db82a8/test/nft.test.ts#L224-L237).

### 1.4 Capture revert data and the failing call index — priority: MEDIUM, size: S-M

- **What walletchan does today**: the inner call's returndata is discarded — `(success, ) = to.call{value}(data)` ([walletchan's contract:112](https://github.com/apoorvlathey/walletchan/blob/470c2767f8c4b35b399b1e97fc00dd9d1ab9a070/apps/contracts/src/utils/TxSimulator.sol#L112)) — and `simulateBatch` continues past failures with only an aggregate `allSuccess` flag ([370-377](https://github.com/apoorvlathey/walletchan/blob/470c2767f8c4b35b399b1e97fc00dd9d1ab9a070/apps/contracts/src/utils/TxSimulator.sol#L370-L377)). A failed preview can say "this will fail" but never *which call* or *why*.
- **Why it matters**: with the raw revert bytes and the failing index, the confirmation UI can show "call 2 of 3 reverts: `InsufficientAllowance(0, 500000000)`" instead of a generic banner. Returndata capture is nearly free (keep the `bytes memory`, return it); decoding `Error(string)`/`Panic(uint256)`/custom selectors is a small TS layer.
- **Adoption note**: two separable pieces — returndata capture (S) and halt-at-first-failure with an index (S). Halting is the honest semantics for the atomic (7821/7702) batches: on-chain, call N+1 never executes after call N reverts, so deltas that include post-failure calls describe a sequence that cannot happen. Continue-past-failure is arguably right for non-atomic keep-broadcasting batches — if kept there, splitting the two semantics keeps each mode honest. Reference: [viem-tx-sim's contract:424-444](https://github.com/frontier159/viem-tx-sim/blob/2db82a8/contracts/TxSimulator.sol#L424-L444) (halt + capture), [simulator.ts:344-364](https://github.com/frontier159/viem-tx-sim/blob/2db82a8/src/internal/simulator.ts#L344-L364) (decode).

### 1.5 Stop trusting batch-gas figures measured past a failed call — priority: MEDIUM, size: S-M

- **What walletchan does today**: `simulateBatchGas` records a figure for every call and only flips `allSuccess` ([walletchan's contract:198-214](https://github.com/apoorvlathey/walletchan/blob/470c2767f8c4b35b399b1e97fc00dd9d1ab9a070/apps/contracts/src/utils/TxSimulator.sol#L198-L214)); the TS layer decodes `allSuccess`, logs it, and applies the 2× buffer to the numbers regardless ([batchGasEstimation.ts:396-424](https://github.com/apoorvlathey/walletchan/blob/470c2767f8c4b35b399b1e97fc00dd9d1ab9a070/apps/extension/src/chrome/batchGasEstimation.ts#L396-L424)).
- **Why it matters**: walletchan's own tier-1 path abandons `eth_simulateV1` entirely when any call reverts, with a comment that states the problem perfectly — a reverted call's `gasUsed` "is the partial gas consumed up to the revert point — NOT what a successful broadcast would use", and trusting it "gives an under-buffered gas limit that OOGs in nested frames" ([batchGasEstimation.ts:257-274](https://github.com/apoorvlathey/walletchan/blob/470c2767f8c4b35b399b1e97fc00dd9d1ab9a070/apps/extension/src/chrome/batchGasEstimation.ts#L257-L274)). Tier 2 has the identical problem, plus one more: calls *after* the failure execute against post-revert state, so for an atomic batch those measurements describe a state sequence that cannot occur on-chain.
- **Adoption note**: cheapest fix at the TS layer — treat `allSuccess == false` from tier 2 the way tier 1 already treats it. Better fix in-contract: break on the first failure and return its index. Reference: [viem-tx-sim's contract:191-210](https://github.com/frontier159/viem-tx-sim/blob/2db82a8/contracts/TxSimulator.sol#L191-L210).

### 1.6 Add the EIP-7623 floor to the calldata gas math — priority: MEDIUM, size: S

- **What walletchan does today**: `simulateBatchGas` adds `21000 + 4·zeroByte + 16·nonZeroByte` so the result is "directly usable as a tx gas limit" ([walletchan's contract:198-226](https://github.com/apoorvlathey/walletchan/blob/470c2767f8c4b35b399b1e97fc00dd9d1ab9a070/apps/contracts/src/utils/TxSimulator.sol#L198-L226)).
- **Why it matters**: since Pectra, EIP-7623 floors every transaction at `21000 + 10·tokens` where `tokens = zeroBytes + 4·nonZeroBytes` — uniformly **2.5×** the standard calldata rate. For calldata-heavy, low-execution calls the floor binds and the 4/16 figure under-states the real charge; the 2× buffer absorbs it only up to roughly 2.6 KB of non-zero (or ~10 KB of zero) calldata on a near-zero-execution call, after which even the buffered limit is below what the chain charges. Data-carrier legs (inscriptions, L2 payload posts, large permit blobs) are the realistic shape.
- **Adoption note**: one `max`: `21000 + max(4z + 16nz, 10·(z + 4nz))`. Reference with exact-value tests including the floor-binding case: [gas.ts:17-27](https://github.com/frontier159/viem-tx-sim/blob/2db82a8/src/internal/gas.ts#L17-L27), [test/gas.test.ts:54-58](https://github.com/frontier159/viem-tx-sim/blob/2db82a8/test/gas.test.ts#L54-L58). Structural suggestion: viem-tx-sim computes this in TypeScript rather than in the contract, precisely so the *next* calldata repricing is a one-line change instead of a bytecode regeneration — an operation walletchan's own docs rightly treat as hazardous (the metadata-hash/`InvalidJump` warning in `_docs/ASSET_CHANGES_SIMULATION.md`).

### 1.7 Verify forged storage slots before trusting them; forge non-max amounts — priority: MEDIUM, size: M

- **What walletchan does today**: the retry path picks the balance/allowance slot as "the token's first access-list storage key that isn't a known EIP-1967 proxy slot" and writes to it unverified ([txSimulation.ts:734-763](https://github.com/apoorvlathey/walletchan/blob/470c2767f8c4b35b399b1e97fc00dd9d1ab9a070/apps/extension/src/chrome/txSimulation.ts#L734-L763)); ERC-20→Permit2 approvals are forged to `MAX_UINT256` ([848-852](https://github.com/apoorvlathey/walletchan/blob/470c2767f8c4b35b399b1e97fc00dd9d1ab9a070/apps/extension/src/chrome/txSimulation.ts#L848-L852)).
- **Why it matters**: for vanilla tokens the first-key heuristic works; for packed slots, multi-slot `balanceOf` (rebasing/shares tokens like stETH, aTokens), or unusual proxies it silently writes a huge number into the *wrong* slot and the retry runs on garbage state — a confident, wrong preview. And exactly-max allowances make standard ERC-20s (and Permit2) skip the allowance decrement, erasing the consumption signal.
- **Adoption note**: one extra `eth_call` per token — write a sentinel to the candidate slot via `stateDiff`, re-read `balanceOf`/`allowance` under that override, accept only on exact match, and *skip the token rather than forge it* if nothing verifies ([probes.ts:50-91](https://github.com/frontier159/viem-tx-sim/blob/2db82a8/src/internal/probes.ts#L50-L91)). viem-tx-sim forges everything with one large non-max sentinel, `10^45` — below `uint256.max` for ERC-20 decrements, below `type(uint160).max` for Permit2's packed amount ([constants.ts:9-24](https://github.com/frontier159/viem-tx-sim/blob/2db82a8/src/constants.ts#L9-L24)).

### 1.8 Permit2: measure the required allowance instead of only forging it — priority: MEDIUM, size: M

- **What walletchan does today**: computes the Permit2 slot and overrides it to `amount = type(uint160).max` with the nonce preserved ([txSimulation.ts:854-896](https://github.com/apoorvlathey/walletchan/blob/470c2767f8c4b35b399b1e97fc00dd9d1ab9a070/apps/extension/src/chrome/txSimulation.ts#L854-L896)) — the nonce-preservation trick viem-tx-sim adopted directly.
- **Why it matters**: max-uint160 has a side effect — Permit2's `AllowanceTransfer` skips the allowance decrement when the stored amount is exactly `type(uint160).max`, so the forged run erases the very signal that would say how much allowance the path *consumes*. With a non-max forge plus a probe, the preview can say "this transaction needs a Permit2 approval of exactly 500 USDC to 0xRouter" instead of "here's what a fully-forged run would do".
- **Adoption note**: forge a large non-max uint160 amount (§1.7's sentinel), then read `allowance(owner, token, spender)` on the singleton around each call inside the simulator ([viem-tx-sim's contract:501-513](https://github.com/frontier159/viem-tx-sim/blob/2db82a8/contracts/TxSimulator.sol#L501-L513)) and sum the per-call decreases, skipping in-batch `approve`/`permit` grants and sanity-clamping against the token's physical outflow ([requirements.ts:163-178](https://github.com/frontier159/viem-tx-sim/blob/2db82a8/src/internal/requirements.ts#L163-L178), [244-278](https://github.com/frontier159/viem-tx-sim/blob/2db82a8/src/internal/requirements.ts#L244-L278)). Bonus hardening from building it: sentinel-verify the computed slot (write, re-read via the getter, require amount match **and** unchanged nonce) before trusting it ([slots.ts:289-327](https://github.com/frontier159/viem-tx-sim/blob/2db82a8/src/internal/slots.ts#L289-L327)).

### 1.9 Per-call checkpoints and gross-outflow tracking — priority: MEDIUM (feature), size: M-L

- **What walletchan does today**: one `balanceOf` sweep before and after the whole batch — a net signed delta per token ([walletchan's contract:314-365](https://github.com/apoorvlathey/walletchan/blob/470c2767f8c4b35b399b1e97fc00dd9d1ab9a070/apps/contracts/src/utils/TxSimulator.sol#L314-L365)).
- **Why it matters**: two things that model can't see, both cheap in the loop already running. **Per-call attribution**: a balance checkpoint after every call (a flat grid of `calls + 1` readings per probe) reconstructs `byCall[i] = row[i+1] − row[i]` with the invariant `sum(byCall) === netDelta` — in a batch UI that turns "USDC −500 overall" into "approve: 0, swap: −500" per row. **Gross vs net**: tracking each token's *minimum* balance across call boundaries gives `maxOutflow = before − min` — the batch's peak exposure. A batch that spends 500 USDC and gets 499 back nets −1 but momentarily hands 500 to a router; for a security-oriented preview, 500 is the number the user should see.
- **Adoption note**: reference machinery at [viem-tx-sim's contract:394-541](https://github.com/frontier159/viem-tx-sim/blob/2db82a8/contracts/TxSimulator.sol#L394-L541) (grids + min-balance tracking) and [checkpoints.ts:13-65](https://github.com/frontier159/viem-tx-sim/blob/2db82a8/src/internal/checkpoints.ts#L13-L65) (reconstruction). The same grid extended to `allowance()` reads is what powers §1.8.

### 1.10 Distinguish "no change" from "couldn't read" in probes — priority: LOW-MEDIUM, size: S

- **What walletchan does today**: in `_computeDeltas`, a candidate whose `balanceOf` reverts reads as 0 on both sides, yields delta 0, and is compacted away ([walletchan's contract:332-365](https://github.com/apoorvlathey/walletchan/blob/470c2767f8c4b35b399b1e97fc00dd9d1ab9a070/apps/contracts/src/utils/TxSimulator.sol#L332-L365)) — indistinguishable from a token that genuinely didn't move. With the (good) 100k probe gas caps, a slow-but-legitimate token lands in this bucket too.
- **Adoption note**: one `bool[]` in the return tuple, ANDed across every read, letting the UI say "couldn't verify token X" instead of implying it didn't change ([viem-tx-sim's contract:515-527](https://github.com/frontier159/viem-tx-sim/blob/2db82a8/contracts/TxSimulator.sol#L515-L527)).

### 1.11 Don't silently gift the account 100,000 ETH — priority: LOW-MEDIUM, size: S

- **What walletchan does today**: every simulation overrides `from`'s balance to `parseEther("100000")` ([txSimulation.ts:1181](https://github.com/apoorvlathey/walletchan/blob/470c2767f8c4b35b399b1e97fc00dd9d1ab9a070/apps/extension/src/chrome/txSimulation.ts#L1181)).
- **Why it matters**: that makes value-carrying previews work for impersonated/watched accounts — but for a *signing* user who genuinely can't fund `value`, the preview shows a clean success for a transaction that will fail. An asset-change preview that can't be wrong about affordability is worth protecting.
- **Adoption note**: simulate with the real balance by default and apply the big override only when needed (watch-only mode, or as a tagged fallback so the UI can badge "balance was assumed"). viem-tx-sim keeps the honest default and makes native forging an explicit opt-in.

### 1.12 Aggregate duplicate ERC-1155 receipts — priority: LOW, size: S

- **What walletchan does today**: receiver hooks append unconditionally (`_pushReceived`, [walletchan's contract:287-299](https://github.com/apoorvlathey/walletchan/blob/470c2767f8c4b35b399b1e97fc00dd9d1ab9a070/apps/contracts/src/utils/TxSimulator.sol#L287-L299)); `_alreadyCaptured` dedups only the two ERC-721 enumeration walks ([443-454](https://github.com/apoorvlathey/walletchan/blob/470c2767f8c4b35b399b1e97fc00dd9d1ab9a070/apps/contracts/src/utils/TxSimulator.sol#L443-L454)). A batch receiving the same ERC-1155 `(collection, id)` in two calls — split fills, claim-then-claim — renders as two rows with split amounts.
- **Adoption note**: one linear scan in the record path — aggregate ERC-1155 amounts per `(token, id)`, no-op repeated ERC-721 ids ([viem-tx-sim's contract:278-292](https://github.com/frontier159/viem-tx-sim/blob/2db82a8/contracts/TxSimulator.sol#L278-L292); [test/nft.test.ts:241](https://github.com/frontier159/viem-tx-sim/blob/2db82a8/test/nft.test.ts#L241)).

### 1.13 Small robustness items — priority: LOW, size: S

- **Classify reverts structurally, not by prose.** Provider revert messages vary; JSON-RPC error code `3` on the error's cause chain is the stable signal (with a `/revert/i` fallback) — geth even returns a partial access list *with* an `error` field, which is still usable. The `.catch(() => [])` fallbacks around `eth_createAccessList` would benefit from distinguishing "call reverted" from "provider refused" ([rpc.ts:143-165](https://github.com/frontier159/viem-tx-sim/blob/2db82a8/src/internal/rpc.ts#L143-L165)).
- **Pin the hand-written ABI to the compiled artifact.** The extension encodes calls against TS-side ABI strings for a contract living in another package; a drift test comparing them to the forge artifact catches silent skew at CI time ([test/abi.test.ts](https://github.com/frontier159/viem-tx-sim/blob/2db82a8/test/abi.test.ts)).

## 2. Judgment calls — differences that are product context, not recommendations

Differences where walletchan's choice is defensible for a wallet even though viem-tx-sim chose otherwise; listed so the omissions above read as deliberate scope, not oversights.

- **Always-on vs opt-in NFT capture**: a wallet always wants NFT rows; a library makes them opt-in so the default path stays lean. (Item 1.2 applies either way.)
- **Baked 2× gas buffer vs pre-buffer output**: shipping a buffered limit is the right product default; a library returns pre-buffer numbers and documents the EIP-150 rationale.
- **`eth_simulateV1` dual-pathing**: the transfer-log merge is genuine added coverage (outgoing ERC-1155) for a product willing to manage per-chain support caching and verdict distrust; a portability-first library stays single-method.
- **String vs typed error vocabulary**: `simulationFailed` + message fits an extension surface; typed error classes fit a library API.
- **Gas-ceiling values** (10M/50M vs 16M + 10M): both empirically motivated; no action either way.
