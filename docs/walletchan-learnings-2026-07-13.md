# Learnings for viem-tx-sim from walletchan (2026-07-13)

Scope: features and behaviors present in **walletchan** but not in **viem-tx-sim** — what each does, why it matters, and how (or whether) viem-tx-sim should adopt it. Companion to `docs/walletchan-comparison-2026-07-13.md` and mirror of `docs/walletchan-upstream-recommendations-2026-07-13.md`.

Sources: **walletchan** — https://github.com/apoorvlathey/walletchan at `470c2767`; **viem-tx-sim** — https://github.com/frontier159/viem-tx-sim at `2db82a8`. Both verified against checked-out source.

Constraints any adoption must honor (viem-tx-sim's pinned invariants): RPC-only with viem as the sole runtime dependency; public `simulate()` emits exactly one `eth_call` and zero `eth_createAccessList`; deterministic candidate/result ordering; the single forged-state sentinel `OVERRIDE_TOKEN_AMOUNT = 10^45` — non-max so ERC-20 and Permit2 allowance decrements stay observable, below `type(uint160).max` so it fits Permit2's packed amount ([constants.ts:9-24](https://github.com/frontier159/viem-tx-sim/blob/2db82a8/src/constants.ts#L9-L24)); transaction reverts as result status, infrastructure failures as typed errors; contract changes imply a gated bytecode regeneration.

## 1. Feature gaps — candidates for adoption

### 1.1 `nextTokenId()` counter walk for non-Enumerable NFT mints — priority: LOW-MEDIUM (demand-gated)

- **What walletchan does**: snapshots `nextTokenId()` per candidate before the call (sentinel `type(uint256).max` if unsupported), then walks the advanced range `[nextBefore, nextAfter)` calling `ownerOf(id)` and records ids owned by `address(this)`, capped at 50 per collection ([TxSimulator.sol:456-511](https://github.com/apoorvlathey/walletchan/blob/470c2767f8c4b35b399b1e97fc00dd9d1ab9a070/apps/contracts/src/utils/TxSimulator.sol#L456-L511)).
- **Why it matters**: it is the only pure-`eth_call` way to identify tokens minted by counter-based ERC-721s that neither fire a receiver hook nor implement Enumerable — Uniswap V4's `PositionManager` is the canonical case. viem-tx-sim's NFT capture (receiver hooks + Enumerable walk) sees such mints only as count deltas, a documented gap ([README](https://github.com/frontier159/viem-tx-sim/blob/2db82a8/README.md)).
- **Adoption note**: pure extension of the existing flag-gated capture machinery — add the counter snapshot next to `_snapshotNftBalances` ([TxSimulator.sol:311-321](https://github.com/frontier159/viem-tx-sim/blob/2db82a8/contracts/TxSimulator.sol#L311-L321)), a `_walkNextTokenId` sibling to `_walkEnumerable` ([326-342](https://github.com/frontier159/viem-tx-sim/blob/2db82a8/contracts/TxSimulator.sol#L326-L342)), both feeding `_recordReceipt` so dedup/aggregation come free; two new selectors routed through the gas-capped `_tryStaticUint` primitive ([548-558](https://github.com/frontier159/viem-tx-sim/blob/2db82a8/contracts/TxSimulator.sol#L548-L558)). No ABI change, no new RPC, rides the same single `eth_call`. **Flags**: bytecode regeneration; the zero-cost-when-unused property must stay test-pinned; `nextTokenId()` is an unstandardized selector (walletchan's own comment scopes it to "Uniswap V4 IPositionManager + many counter-based ERC-721s"), with the `ownerOf` check filtering false positives. Adopt when a consumer needs V4 position identity — additive scope against a documented limitation, not a correctness gap.

### 1.2 ERC-7821 self-call access list for cross-call dependency discovery — priority: LOW

- **What walletchan does**: for atomic batches, encodes the whole batch as an ERC-7821 `execute(mode, executionData)` self-call and runs **one** `eth_createAccessList` over it ([txSimulation.ts:1979-2013](https://github.com/apoorvlathey/walletchan/blob/470c2767f8c4b35b399b1e97fc00dd9d1ab9a070/apps/extension/src/chrome/txSimulation.ts#L1979-L2013)) — the trace sees post-call-1 state, capturing addresses only reachable after an earlier call's state change. Falls back to N per-call lists when `from` is not an on-chain 7821 executor.
- **Why it matters**: viem-tx-sim's discovery always traces N per-call lists against pre-batch state, so cross-call-dependent addresses can be missed (documented limitation, [README](https://github.com/frontier159/viem-tx-sim/blob/2db82a8/README.md); [queryDiscovery.ts:32-45](https://github.com/frontier159/viem-tx-sim/blob/2db82a8/src/internal/queryDiscovery.ts#L32-L45)).
- **Adoption note**: transfers poorly to a generic library — the trick requires `from` to actually be an on-chain ERC-7821 account at trace time; for a plain EOA the self-call reverts and the fallback is what viem-tx-sim already does. It would also change discovery's pinned RPC shape (1 vs N access lists) and import an execution-mode encoding into an account-model-agnostic library. Shape if ever: an opt-in discovery strategy for consumers whose accounts qualify. Keep the blind spot documented until such a consumer materializes.

### 1.3 EIP-7702 delegate-code awareness in gas estimation — priority: LOW (docs-first)

- **What walletchan does**: when an EOA will be 7702-delegated at broadcast, fetches the delegate's runtime code and applies it as the state override for `eth_estimateGas` ([gasEstimation.ts:205-211](https://github.com/apoorvlathey/walletchan/blob/470c2767f8c4b35b399b1e97fc00dd9d1ab9a070/apps/extension/src/chrome/gasEstimation.ts#L205-L211), [287-323](https://github.com/apoorvlathey/walletchan/blob/470c2767f8c4b35b399b1e97fc00dd9d1ab9a070/apps/extension/src/chrome/gasEstimation.ts#L287-L323)), plus a flat per-authorization-tuple gas bump because estimation never sees the authorization list that broadcast attaches.
- **Why it matters**: viem-tx-sim's `gas.estimateBatch` injects the ghost at `from`, *replacing* any delegate code — so for a batch that will execute through a 7702 delegate's dispatch on-chain, the measurement excludes the delegate's routing overhead and the per-tuple authorization cost. Real but bounded error, same direction as the documented warm/cold under-measurement.
- **Adoption note**: one README error-bar bullet now; a "measure through supplied wrapper code" mode conflicts with the ghost-at-`from` design and isn't worth designing unasked.

### 1.4 Non-standard gas-model chain awareness — priority: LOW (docs-only)

- **What walletchan does**: skips both simulation-based gas tiers on chains flagged `usesNonStandardGasModel` — MegaETH's separate storage-gas dimension is invisible to `gasleft()`-based measurement ([batchGasEstimation.ts:136-163](https://github.com/apoorvlathey/walletchan/blob/470c2767f8c4b35b399b1e97fc00dd9d1ab9a070/apps/extension/src/chrome/batchGasEstimation.ts#L136-L163)).
- **Why it matters**: `gas.estimateBatch` has the same blindness and doesn't say so.
- **Adoption note**: one README limitation line ("`gasleft()`-based measurement assumes the standard EVM gas model"). No per-chain registry — chain knowledge is a product concern, not a library one.

## 2. Deliberate non-adoptions — walletchan behaviors viem-tx-sim intentionally omits

- **`eth_simulateV1` as a second simulation channel** ([txSimulation.ts:2189-2428](https://github.com/apoorvlathey/walletchan/blob/470c2767f8c4b35b399b1e97fc00dd9d1ab9a070/apps/extension/src/chrome/txSimulation.ts#L2189-L2428)). The strongest candidate, so argued in full: over access-list discovery it adds exactly two things — visibility of ERC-1155/outgoing transfers via `traceTransfers` logs, and the cross-call-dependency coverage of §1.2. Against that: a second RPC method breaks the single-method portability story and the pinned RPC counts; provider support is inconsistent enough that walletchan maintains a 10-minute per-chain support cache ([txSimulation.ts:406-417](https://github.com/apoorvlathey/walletchan/blob/470c2767f8c4b35b399b1e97fc00dd9d1ab9a070/apps/extension/src/chrome/txSimulation.ts#L406-L417)) — statefulness a library shouldn't own; and walletchan's own production experience required hard-coding distrust of its verdict after Alchemy false-positive-reverted valid approve+swap batches ([txSimulation.ts:2485-2519](https://github.com/apoorvlathey/walletchan/blob/470c2767f8c4b35b399b1e97fc00dd9d1ab9a070/apps/extension/src/chrome/txSimulation.ts#L2485-L2519); the same distrust independently in [batchGasEstimation.ts:257-274](https://github.com/apoorvlathey/walletchan/blob/470c2767f8c4b35b399b1e97fc00dd9d1ab9a070/apps/extension/src/chrome/batchGasEstimation.ts#L257-L274)). A caller wanting v1-log-derived queries can build them on viem in ~30 lines and feed `balanceQueries` directly. Reopen trigger: concrete demand for ERC-1155 balance modeling — the one job where log-derived ids beat every contract-side alternative; if reopened, a separate discovery-only helper that never feeds `simulate()` and never supplies a verdict.
- **Automatic 100,000-ETH balance override at `from`** ([txSimulation.ts:1181](https://github.com/apoorvlathey/walletchan/blob/470c2767f8c4b35b399b1e97fc00dd9d1ab9a070/apps/extension/src/chrome/txSimulation.ts#L1181)). Makes every preview lie about native affordability; viem-tx-sim keeps the honest default and offers opt-in `nativeBalanceOverrides`.
- **Automatic retry-with-forged-state inside the simulate path** ([txSimulation.ts:1115-1128](https://github.com/apoorvlathey/walletchan/blob/470c2767f8c4b35b399b1e97fc00dd9d1ab9a070/apps/extension/src/chrome/txSimulation.ts#L1115-L1128)). A conditional second `eth_call` plus per-candidate discovery calls breaks the one-`eth_call` pin and makes RPC cost unpredictable; forging stays an explicit, separately-billed step (`tokenOverrides.*`).
- **Unverified slot discovery** ([txSimulation.ts:734-763](https://github.com/apoorvlathey/walletchan/blob/470c2767f8c4b35b399b1e97fc00dd9d1ab9a070/apps/extension/src/chrome/txSimulation.ts#L734-L763)). Writing to the first non-proxy access-list key silently forges garbage state on packed/multi-slot/unusual layouts; sentinel-write-and-readback verification ([probes.ts:50-91](https://github.com/frontier159/viem-tx-sim/blob/2db82a8/src/internal/probes.ts#L50-L91)) is strictly safer, with `unresolved` for the rest.
- **`MAX_UINT256` allowance forging** ([txSimulation.ts:848-852](https://github.com/apoorvlathey/walletchan/blob/470c2767f8c4b35b399b1e97fc00dd9d1ab9a070/apps/extension/src/chrome/txSimulation.ts#L848-L852)). Max suppresses both ERC-20 and Permit2 allowance decrements — exactly the signal requirement measurement observes; the non-max `10^45` sentinel exists to preserve it, and max is rejected with a typed error ([simulator.ts:411-431](https://github.com/frontier159/viem-tx-sim/blob/2db82a8/src/internal/simulator.ts#L411-L431)).
- **Continue-past-failure batch execution** ([TxSimulator.sol:370-377](https://github.com/apoorvlathey/walletchan/blob/470c2767f8c4b35b399b1e97fc00dd9d1ab9a070/apps/contracts/src/utils/TxSimulator.sol#L370-L377)). Forfeits `failingCallIndex`/`revertData` and would break the zero-tail `byCall` invariant; for atomic batches it describes a sequence that cannot occur on-chain. Genuine keep-broadcasting semantics would be a new explicit execution mode, not a change to the default.
- **Any-signer ERC-1271** ([TxSimulator.sol:565-583](https://github.com/apoorvlathey/walletchan/blob/470c2767f8c4b35b399b1e97fc00dd9d1ab9a070/apps/contracts/src/utils/TxSimulator.sol#L565-L583)). Accepting any recoverable signature validates forged permits during simulation; signer-bound recovery ([TxSimulator.sol:212-214](https://github.com/frontier159/viem-tx-sim/blob/2db82a8/contracts/TxSimulator.sol#L212-L214)) reproduces real EOA verification.
- **ERC-7715 static decode, metadata/pricing enrichment, retry loops, post-confirmation log extraction, per-chain gas-model registry.** Wallet-product concerns (UI freshness, CoinGecko, chain registry); a library consumer composes these, and none affect simulation correctness at this layer.
